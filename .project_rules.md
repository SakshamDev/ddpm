# DDPM Project — Operating Rules & Workflow

> **Purpose:** Persistent reference for the 13 non-negotiable operating rules  
> and the 10-step per-module workflow. These survive across sessions even if  
> the original prompt scrolls out of context.

---

## 13 Operating Rules

1. **Smallest testable unit.** Never generate a full module in one shot. Generate the smallest testable unit, validate it, then proceed.
2. **Teach before coding.** For every new concept/module: problem → intuition → math → tensor shapes → common mistakes — *before* implementation.
3. **Validate before scaling.** Nothing gets wired into the full U-Net or pipeline until it passes unit tests in isolation, on CPU, on tiny/synthetic input.
4. **No GPU time on unvalidated code.** Nothing touches Colab until it has passed local CPU tests.
5. **Always show tensor shapes.** At every point a tensor is created, reshaped, or transformed — in both code comments and prose.
6. **Understanding checkpoints.** Ask the user to restate each concept in their own words. Log the restatement in `docs/concepts.md` before moving on.
7. **Explain the *why*.** Especially for non-default design choices (ε-prediction, GroupNorm, attention only at 16×16, etc.).
8. **One change at a time.** When debugging or extending. Log every experiment in `experiments/exp_log.md`.
9. **Checkpoint discipline from session 1.** Save/resume must work before any long run.
10. **Propose the cheaper fallback up front.** Don't wait for OOM to suggest smaller batch, fewer channels, or gradient accumulation.
11. **No silent scope creep.** If a request would skip ahead of the current milestone, pause and confirm.
12. **State uncertainty plainly.** If not confident about a detail, say so rather than asserting.
13. **Plan before code.** Use Implementation Plan / Task List artifacts for every milestone. Wait for explicit user approval before writing code.

## 10-Step Per-Module Workflow

For each module (noise schedule, forward process, time embedding, ResBlock, attention, U-Net, training loop, sampler, etc.):

1. Explain the **problem** this module solves.
2. Explain the **intuition** in plain language.
3. Explain the **math**, with shapes annotated.
4. Explain the exact **tensor shapes** flowing through it.
5. Explain **common implementation mistakes** specific to this module.
6. Generate the **implementation**.
7. Generate **unit tests**.
8. Generate a **visualization or sanity-check output** where applicable.
9. **Verify correctness** against the tests/visualization before declaring it done.
10. **Log** the concept in `concepts.md` and any non-default decision in `decisions.md`.

Only then move to the next module.

## Pre-Milestone Paper Analysis

Before any milestone begins:

1. Read and summarize the relevant paper section(s).
2. Explain what the original authors were trying to solve.
3. Explain what alternative approaches existed at the time.
4. Explain why the paper's solution was novel.
5. Identify which parts we will reproduce exactly and which parts we will simplify.

No implementation begins until this analysis is complete.

## Colab + T4 Constraints

- T4, 16 GB VRAM, free-tier sessions can disconnect at any time.
- All checkpoints save to Google Drive (mounted), never to Colab local disk.
- Checkpoint contents: model state_dict, EMA state_dict, optimizer state_dict, step, config, seed.
- Colab notebook is a thin shell: clone/pull → install deps → mount Drive → auto-resume → run.
- Confirm fp32 correctness before enabling AMP/fp16.

## Target Architecture Summary

- ~35M params
- Base channels: 128
- Channel multipliers: [1, 2, 2, 2]
- Self-attention: 16×16 only
- ResBlocks per resolution: 2
- T = 1000 timesteps
- Cosine noise schedule
- ε-prediction
- Classifier-free guidance via label dropout (single training run)
- ~200K training steps, batch size 128
- Target FID < 10
